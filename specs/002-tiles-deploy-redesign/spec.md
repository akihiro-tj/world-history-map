# Feature Specification: タイル配信アーキテクチャの再設計（packages/tiles 化・原子的デプロイ・DEV/PROD R2 分離）

**Feature Branch**: `002-tiles-deploy-redesign`
**Created**: 2026-04-25
**Status**: Draft
**Input**: User description: "タイル配信アーキテクチャを「packages/tiles 化 + manifest ビルド時埋め込み + DEV/PROD R2 分離」に再設計する"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - タイル更新を 1 つの git 操作で原子的にデプロイする (Priority: P1)

データ更新者は、地図タイルを更新したいときに「ローカルでデータ生成 → コミット → プッシュ」だけで完了させたい。
本番環境では `manifest`（年 → ファイル名のマッピング）が frontend バンドルに焼き込まれ、frontend のデプロイと R2 の参照対象が常に同じバージョンで一致する。
これにより「manifest だけ更新を忘れて本番が壊れる」「先に manifest が公開されて存在しないファイルを参照する」「最大 10 分の伝搬ラグ中にユーザーが不整合状態を見る」などの事故が起きない。

**Why this priority**: 現状の最大の事故源（手動 2 コマンド・runtime fetch ラグ・順序依存破壊）を根本から解消する。これだけで MVP として価値がある。

**Independent Test**: タイルを 1 年分更新し、PR を作成・マージしてデプロイした際、フロントエンドが新タイルを参照する瞬間と R2 がそのタイルを返せる状態になる瞬間が**完全に同一**で、不整合の窓が 0 秒であることを観測できる。

**Acceptance Scenarios**:

1. **Given** 1600 年のタイルデータが更新されローカルでタイル生成が完了している、**When** データ更新者が `git commit && git push` を行い PR が main にマージされる、**Then** フロントエンドが新しい 1600 年タイルを参照しはじめると同時に、R2 にもそのタイルが既に存在しており、ユーザーは一切の空タイル / エラーを経験しない
2. **Given** 過去のデプロイで R2 と frontend が独立に動いていた状態、**When** 新アーキテクチャに移行した、**Then** タイル更新フローは「複数手動コマンドの組合せ」から「単一の git ワークフロー」に集約される
3. **Given** タイル更新を含む PR、**When** タイル binary のアップロードが何らかの理由で失敗する、**Then** フロントエンドの本番デプロイは進行せず、本番が壊れた状態で公開されることはない

---

### User Story 2 - PR ごとのプレビューで本番影響なくタイルを検証する (Priority: P2)

レビュアーは、PR の動作確認をするときに、新しいタイルを含むプレビュー環境にアクセスして見た目と挙動を確認したい。
本番ユーザーへの影響は一切なく、本番マージ前に視覚的検証ができる。

**Why this priority**: 検証なしで本番に出す現状を改善し、データ品質事故を未然に防ぐ。P1 の上に乗る形で価値を増す。

**Independent Test**: 1 つの PR をオープンして、本番とは独立した URL でその PR のタイルが表示されることを確認できる。本番のフロントエンドからは依然として旧タイルが見える。

**Acceptance Scenarios**:

1. **Given** タイル更新を含む PR が新規にオープンされた、**When** CI が完了する、**Then** 本番ドメインとは別のプレビュー URL でその PR のタイル変更を含む地図が閲覧できる
2. **Given** プレビューで動作確認している、**When** 同時に本番ユーザーがアクセスする、**Then** 本番ユーザーには変更前のタイルが表示され、プレビューの存在を一切認識しない
3. **Given** 同じ PR を複数回 push する、**When** 各 push で CI が走る、**Then** プレビューは新しい push の内容に追従し、本番には影響しない

---

### User Story 3 - 不正データを git revert だけでロールバックする (Priority: P2)

運用者は、タイル更新後に問題が発覚したとき、対応するコミットを `git revert` するだけで本番が安全に元の状態に戻ることを期待する。
追加の手動操作（manifest を手書きで戻す、R2 にファイルをアップロードし直す）は不要。

**Why this priority**: 事故発生時の MTTR を最小化する。発生頻度は P1/P2 より低いが、起きたときの被害は大きいため上位に置く。

**Independent Test**: 任意のタイル更新コミットを git revert し、自動デプロイが完了した後、フロントエンドが revert 前の状態で正しく表示されることを確認できる。手動の R2 操作は一切不要。

**Acceptance Scenarios**:

1. **Given** タイル更新コミット X が main にマージされ本番デプロイ済み、**When** コミット X を revert する PR をマージする、**Then** フロントエンドの再デプロイ完了とともに、本番は X 以前の状態に戻る
2. **Given** ロールバック後、**When** ユーザーが地図を表示する、**Then** revert 対象の年のタイルは旧バージョンの内容を返し、空タイルやエラーは発生しない
3. **Given** ロールバックの判断から実行までの時間、**When** 計測する、**Then** revert PR 作成からフロントエンドの本番反映完了まで通常のデプロイ時間内（5 分以内）に収まる

---

### User Story 4 - 古いタイルファイルを自動掃除してストレージを抑える (Priority: P3)

運用者は、ロールバック用に古いタイルを残しつつ、無制限にストレージが膨らまないように、一定の世代より古いタイルが自動削除される仕組みを期待する。

**Why this priority**: コスト面の課題で、即座の機能破綻には至らないが、長期運用で必ず顕在化する。P1〜P3 を実現した後の運用最適化として独立に追加できる。

**Independent Test**: 直近 N 世代のコミットで参照されているタイルファイル一覧と R2 上の実ファイル一覧を突合し、参照されていない古いファイルだけが削除されることを観測できる。

**Acceptance Scenarios**:

1. **Given** ロールバック窓 N コミット分の参照タイル一覧が取得できる、**When** GC を実行する、**Then** その N コミットのいずれにも参照されないタイルファイルだけが R2 から削除される
2. **Given** GC が実行された直後、**When** 直近 N コミットのいずれかのバージョンへロールバックする、**Then** 必要なタイルは依然として R2 に存在し、ロールバックは成功する
3. **Given** GC の対象範囲外（より古い）コミットへロールバックを試みる、**When** ファイルが既に削除されている、**Then** 事前に「ロールバック窓を超えている」ことが検知でき、デプロイが本番反映前に停止する

---

### Edge Cases

- **ハッシュ計算の非決定性**: タイル生成ツール（tippecanoe）の出力は環境差でバイト単位がズレうる。これにより同じ上流データから別々の開発者が生成すると違うハッシュが出る可能性がある → ハッシュ計算は「git にコミットされた binary」を入力源とすることで、ハッシュは binary の内容にのみ依存し、生成環境に依存しない
- **長期間放置された PR のマージ**: PR 作成から数週間後にマージしようとすると、その間に GC が走って DEV bucket から該当タイルが消えている可能性がある → CI が再 push 時に存在チェック → 不在ならアップロード、で復旧可能
- **タイル生成の途中失敗による中途半端な dist**: 一部年だけ新ハッシュで他は旧ハッシュ、という状態のコミットを許すか → コミット時の事前検証（manifest と dist の整合性チェック）でブロックする
- **DEV と PROD で同じ binary が存在保証されるか**: DEV にだけアップして PROD には未到達の状態でフロントエンドが PROD を見ると 404 → main マージ時の CI で PROD アップロード成功を frontend 本番デプロイのゲートにする
- **同年タイルへの並行 PR のマージコンフリクト**: 2 PR が同じ年の `world_{year}.pmtiles` を更新するとバイナリ衝突。最後に解決されたほうのハッシュが manifest に残る → 通常の git コンフリクト解決の対象とし、特別な機構は要らない
- **wrangler / Cloudflare API トークンの漏洩・失効**: CI が認証失敗してアップロードできない → 手動デプロイ経路を完全に廃止せず、緊急時のフォールバックとしてローカル wrangler を使える状態は残す
- **本番フロントエンドの古いブラウザキャッシュ**: 旧 frontend バンドル（旧 manifest 埋め込み）を保持しているクライアントは旧ハッシュを引き続き参照する → 旧ハッシュは GC 窓内であれば R2 に残るので問題ない。GC 窓はブラウザ最大キャッシュ寿命より長く設定する

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: タイル更新を本番に反映するワークフローは、単一の git 操作（commit + push + マージ）に集約される。複数の手動コマンドや手動の認証情報投入を必要としない
- **FR-002**: フロントエンドが参照するタイルファイル名のマッピング情報は、フロントエンドのビルド成果物に同梱される。実行時にネットワーク経由で別途取得する設計を取らない
- **FR-003**: フロントエンドのバージョンと R2 上のタイル参照は、常に同じバージョン識別子（コミット）に紐づき、伝搬ラグなく同時に切り替わる
- **FR-004**: タイル binary はハッシュベースのファイル名で識別され、同名ファイルが異なる内容を持つことはない（immutable）
- **FR-005**: ハッシュ計算は git に保存された binary の内容に対して決定的に行われ、生成した開発者の環境によって結果が変わらない
- **FR-006**: PR をオープンすると、その PR のタイル変更を含む独立したプレビュー環境が用意される。プレビューは本番ユーザーから到達不能でなければならない
- **FR-007**: タイル binary のアップロード先は、本番用ストレージとプレビュー用ストレージで分離される。プレビュー作業が本番ストレージの内容に影響を与えてはならない
- **FR-008**: タイル binary のアップロードは、CI 環境に保管された認証情報のみで完結する。開発者ローカルに本番認証情報を保持する必要があってはならない
- **FR-009**: タイル binary のアップロードは差分のみを行い、既にストレージ上に存在するハッシュ付きファイルを再アップロードしない
- **FR-010**: 不正なタイル更新コミットを `git revert` した PR をマージすると、フロントエンドの自動再デプロイによって本番が revert 前の状態に戻る。手動の R2 操作を要しない
- **FR-011**: 直近 N コミット分（ロールバック窓）の manifest が参照するタイル binary は、R2 から削除されない
- **FR-012**: ロールバック窓 N より古いコミットでのみ参照される binary は、定期的に R2 から自動削除される（GC）
- **FR-013**: タイルを変更しないコード変更（フロントエンド UI のみの変更等）に対しては、タイル binary の差分アップロード step は **実行されず skip される**（ワークフロー本体は走るが R2 への書き込みは発生しない）。なお、**フロントエンド本番デプロイ自体は変更内容に依らず main マージごとに必ず起動する**（タイル変更なしの main マージでフロントエンドが反映されない事態を起こさない）
- **FR-014**: タイル binary を生成するローカル処理（pipeline run）の責務と、ハッシュ計算 / マニフェスト生成の責務は明確に分離され、後者は git に保存済みの binary だけを入力とする
- **FR-015**: 既存のテリトリ説明データ（descriptions, index.json, color-scheme.json）の更新フローは、本変更の影響を受けない
- **FR-016**: 本変更の前後で、フロントエンドが提供するエンドユーザー機能（地図表示・年選択・領土クリック等）の挙動に差異が生じない
- **FR-017**: 移行作業中も、本番ユーザーがダウンタイムやエラー画面を経験しない（旧アーキテクチャから新アーキテクチャへの切り替えは破壊的でない）

### Key Entities

- **タイルパッケージ**: ハッシュなしの「マスター binary（年単位）」と、ハッシュ付きの「配信成果物 + マッピング情報」を提供するワークスペース内部資産。フロントエンドが直接 import して利用する
- **マスター binary**: git 管理下に置かれる、年単位 1 ファイルの上書き保存される PMTiles。tippecanoe の出力をそのまま反映する
- **配信成果物**: マスター binary から決定的に派生する、ハッシュ付きファイル名の immutable ファイル。R2 にアップロードされる
- **マッピング情報（manifest）**: 年 → 配信成果物のファイル名の対応表。フロントエンドにビルド時に埋め込まれる
- **配信ストレージ（DEV/PROD）**: 配信成果物を保管する 2 系統のストレージ。プレビュー用と本番用が分離される
- **バージョン識別子**: 本仕様におけるタイル状態の単位。git コミットがそのまま識別子となる
- **ロールバック窓**: GC で残す最近のコミット世代数。この窓内のコミットへのロールバックは追加操作なしで成立する

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: タイル更新後にエンドユーザーが「manifest と binary の不一致による空タイル / エラー」を経験する事象が **発生しなくなる**（観測可能なエラー率が 0%）
- **SC-002**: タイル更新を本番に反映するために実行する手動コマンド数が、現状の **2 コマンド以上**（pipeline run + publish-manifest）から **0 コマンド**（git push のみ、CI が自動実行）に削減される
- **SC-003**: 不正データに気付いてからフロントエンド本番が正しい状態に戻るまでの所要時間が **5 分以内**（revert PR のマージから自動デプロイ完了まで）
- **SC-004**: PR をオープンしてから、その変更を含むプレビュー環境にアクセス可能になるまでの時間が **5 分以内**
- **SC-005**: R2 ストレージの占有量が、過去 N コミット分（ロールバック窓）で参照されるタイル数に **線形に比例**し、上限が予測可能（時間経過に対して無制限に増えない）
- **SC-006**: 開発者がローカルに Cloudflare 本番認証情報を保持しない状態で、タイル更新の本番反映が可能になる（CI が代行）
- **SC-007**: 移行完了後、初回訪問〜再訪問のいずれの場面でも、フロントエンドが追加で manifest を実行時取得するためのネットワーク往復は **0 回**
- **SC-008**: 任意の PR について、本番ユーザーがその PR の変更を本番デプロイ前に観測する確率が **0%**

## Assumptions

- 本リポジトリは既に pnpm monorepo 構成であり、新規ワークスペースパッケージの追加は通常の手続きで可能である
- Cloudflare R2 / Pages / Workers のサービスを今後も使用する。R2 の追加バケット作成権限がプロジェクトオーナーにある
- GitHub Actions に Cloudflare の API トークンを Secret として保管する手段が利用可能である
- ロールバック窓 N の初期値は 3 を想定する。**単位は「`packages/tiles/src/manifest.ts` が変化した main コミット」**（タイルを変更しない main コミットは N にカウントされない）。運用しながら調整可能
- 一般ブラウザクライアントの最終訪問〜再訪問の最大間隔は数日のオーダーであり、ロールバック窓（コミット数換算）はこれを十分カバーする頻度で更新される前提
- territory descriptions の更新フロー（Notion 同期 → git commit → Pages デプロイ）は本仕様のスコープ外であり、現状維持
- Cloudflare Pages の preview deployment が本リポジトリで利用可能で、PR 単位で独立 URL を発行する標準機能を活用できる
- 移行期間中は旧 Worker の `/manifest.json` エンドポイントを当面残し、新フロントエンドへの移行完了後にエンドポイントを削除する段階移行を取れる

## Out of Scope

- territory-sync（Notion 同期）の再設計
- Worker のロールバック自動化（Cloudflare 標準機能の利用方針整理は含めるが、自動化機構は構築しない）
- R2 バケットの versioning 機能の有効化（本仕様の GC で代替）
- tippecanoe 自体のバイト決定的な出力への置換
- マスター binary を Git LFS 化する変更（現状の通常 git 管理を継続）
- フロントエンドのエンドユーザー機能（地図 UI / 領土詳細パネル等）の変更

## Dependencies

- 上流リポジトリ `aourednik/historical-basemaps` の継続的な利用可能性
- Cloudflare R2 / Pages / Workers サービスの稼働
- pnpm workspace 機能
- GitHub Actions の Secret / OIDC 機能
- 既存の `pipeline run` でのタイル生成プロセス（責務移管後も引き続き使用）

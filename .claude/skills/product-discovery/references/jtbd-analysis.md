# Jobs-to-Be-Done (JTBD) 分析ガイド

## 概念

JTBD はユーザーが「雇う」機能ではなく、達成したい「ジョブ（目的）」に焦点を当てるフレームワーク。
「ユーザーはドリルが欲しいのではなく、穴を開けたい」という視点でプロダクトを分析する。

## ジョブの構造

ジョブは以下の 3 層で捉える:

| 層 | 説明 | 例（歴史地図の場合） |
|---|---|---|
| Core Job | ユーザーが根本的に達成したいこと | 世界史の空間的な変遷を理解する |
| Related Jobs | Core Job に付随して発生するジョブ | 特定の時代を深掘りする、他者と知識を共有する |
| Emotional Jobs | 感情面で達成したいこと | 知的好奇心を満たす、発見の喜びを得る |

## コードベースからのジョブ特定手順

### Step 1: ユーザーフローの列挙

UI コンポーネント・ルーティング・イベントハンドラを走査し、ユーザーが実行可能なアクションを一覧化する。

確認対象:
- `apps/frontend/src/` 配下のコンポーネント構造
- ルーティング定義（URL パス）
- クリック/ホバー/入力等のイベントハンドラ
- 状態管理（何をトラッキングしているか）

### Step 2: アクションからジョブへの変換

各アクションに対して「ユーザーはなぜこのアクションを実行するのか？」を問い、ジョブとして言語化する。

変換ルール:
- 動詞 + 目的語 + 文脈修飾語の形式で記述する
- 例: 「年を選択する」→「特定の時代の領土状況を確認するために年を選択する」

### Step 3: ジョブマップの作成

ジョブを時系列フローで整理する:

1. **Define** — ジョブの目標を定義する（何を知りたいか決める）
2. **Locate** — 必要な情報を見つける（年代・地域を特定する）
3. **Prepare** — 情報を取得する準備をする（地図を適切な状態にする）
4. **Execute** — 情報を取得する（地図を閲覧する、詳細を読む）
5. **Monitor** — 結果を確認する（得た情報が正しいか検証する）
6. **Modify** — 探索を調整する（別の年代・地域に移動する）
7. **Conclude** — ジョブを完了する（知識を得て離脱する、共有する）

### Step 4: 充足度評価

各ジョブについて、現在のプロダクトがどの程度充足しているかを評価する。

| 充足度 | 定義 |
|--------|------|
| Over-served | 必要以上に複雑な実装がある |
| Well-served | 適切に機能している |
| Under-served | 機能はあるが不十分 |
| Unserved | 対応する機能がない |

**改善機会は Under-served と Unserved のジョブにある。**

## 出力フォーマット

```markdown
| ジョブ | 層 | ジョブマップ段階 | 充足度 | 根拠 |
|--------|-----|-----------------|--------|------|
| ... | Core/Related/Emotional | Define〜Conclude | Over/Well/Under/Unserved | ... |
```

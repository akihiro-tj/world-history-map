# Current Spec テンプレート

`specs/current/<topic>.md` は以下の構成に従う。トピックに該当しないセクションは省略してよい。

## 共通ルール

- バージョン番号、コマンド一覧、ファイルの網羅的リストは書かない
- Mermaid 図で概要を伝え、テキストは補足に留める
- 1ファイル 100 行以内を目安とする
- 型定義はコードから正確にコピーする（意訳しない）

---

## テンプレート

```markdown
# <トピックタイトル>

> Last synced: YYYY-MM-DD

## 概要

スコープと目的を1〜2文で要約する。

## <メイン図>

Mermaid 図でトピックの核心を示す。トピックによって適切な図を選ぶ:
- data-flow: データの変換チェーンを示すフロー図
- data-model: 型の関連と変換を示す図
- frontend-state: 状態の流れとコンポーネントの関係図

## 境界と変換ポイント

アプリ間・レイヤー間の境界でデータがどう変換されるかを記述する。
コード内部の詳細ではなく、境界をまたぐ部分に焦点を当てる。

## 設計判断（任意）

コードからは読み取れない非自明な判断がある場合のみ記録する:

| 判断 | 根拠 | 出典 |
|------|------|------|
| ... | ... | `specs/00X-.../file.md` or コード慣習 |
```

## トピック別ガイドライン

### data-flow

核心: Notion → Pipeline → R2 → Worker → Frontend のデータ変換チェーン

書くべきこと:
- 各境界でのデータ形式の変換（例: Notion Page → GeoJSON → PMTiles）
- 境界をまたぐ通信方法（API, ファイル I/O, HTTP）
- データの流れの全体像を示す Mermaid フロー図

書かないこと:
- 各アプリ内部の実装詳細
- デプロイ方法やインフラ構成

### data-model

核心: ドメインモデルの型定義と、アプリ間での変換関係

書くべきこと:
- ドメインの中核となる型定義（Territory, Period 等）をコードからそのまま引用
- 同じ概念が各アプリでどの型で表現されるか（変換チェーン）
- 型の関連を示す Mermaid 図

書かないこと:
- UI コンポーネントの props 型
- ユーティリティ型
- 全 export 型の網羅的リスト

### frontend-state

核心: UI の状態設計と、ユーザー操作に対するデータの流れ

書くべきこと:
- 状態の管理方法（URL params, Context, MapLibre 内部状態など）と役割分担
- 主要なユーザー操作フロー（年代選択 → レイヤー更新など）を Mermaid で図示
- 状態間の同期ポイント

書かないこと:
- 個々のコンポーネントの props 一覧
- CSS / スタイリングの詳細

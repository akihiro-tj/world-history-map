# Current Spec テンプレート

`specs/current/<topic>.md` は以下の構成に従う。トピックに該当しないセクションは省略してよい。

---

## テンプレート

```markdown
# <トピックタイトル>

> Last synced: YYYY-MM-DD

## 概要

スコープと目的を1〜2段落で要約する。

## アーキテクチャ

Mermaid 図で全体構造を示す。

```mermaid
graph TD
  ...
```

## 主要コンポーネント

| コンポーネント | パス | 役割 |
|--------------|------|------|
| ... | `path/to/file.ts` | ... |

## データフロー

主要ユースケースのデータフローを記述する。必要に応じて Mermaid のシーケンス図やフローチャートを使う。

## 型定義

コードから正確に抽出した型定義を記載する。TypeScript コードブロックを使う:

```typescript
interface Example {
  id: string;
  name: string;
}
```

## 設計判断

コードからは読み取れない非自明な判断を記録する。アーカイブに根拠がある場合は出典を明記する:

| 判断 | 根拠 | 出典 |
|------|------|------|
| ... | ... | `specs/00X-.../file.md` or コード慣習 |
```

---

## セクション別ガイドライン

### 概要
- モジュール/アプリの責務と境界を明記する
- 主要技術を記載する

### アーキテクチャ
- Mermaid 図で視覚的に構造を示す
- モジュール境界と依存関係を表現する
- `architecture` トピック: モノレポ全体の構造を示す

### 主要コンポーネント
- 重要なモジュール/ファイルを役割とともにリストする
- 実際のファイルパスを使う
- `frontend` トピック: コンポーネント、hooks、contexts を含める
- `data-pipeline` トピック: ステージを実行順にリストする

### データフロー
- 主要ユースケースの正常系を記述する
- データ形式の変換を含める（例: GeoJSON → PMTiles）
- `frontend` トピック: ユーザー操作 → 状態変更 → 描画

### 型定義
- コードから正確にコピーする（意訳しない）
- エクスポートされた型のみ含める
- 関連する型をグループ化する
- `data-model` トピック: これが主要セクションになる

### 設計判断
- "What" ではなく "Why" に焦点を当てる
- アーカイブ（`specs/00X-*`）に根拠がある場合は引用する
- コードからは明らかでない制約を含める

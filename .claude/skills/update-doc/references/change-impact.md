# 変更影響マップ

コード変更がどのドキュメントに影響するかの判定基準。
git log で拾ったコミットの diff を見て、対応するドキュメントに反映すべきかを判断する材料にする。

## 反映すべき変更の型

### overview.md に影響するもの（全体の構造変化）

- アプリの追加・削除・名称変更
- アプリ間データフローの変更（GeoJSON → PMTiles → R2、Notion → JSON など）
- 中核の型のフィールド変化（複数アプリで共有されているため）
- `public/data/` 配下の JSON ファイルの種類の増減

### pipeline.md に影響するもの

- ステージの追加・削除・順序変更
- 外部データソースの追加・変更・削除（GitHub リポジトリ、Notion DB など）
- 出力先の変更（R2 バケット、`public/data/` 配下）
- 外部ツール依存の変更（tippecanoe, wrangler など）
- 増分処理ロジックの変更（ハッシュの取り方、キャッシュ構造）
- CLI サブコマンドの追加・削除

### frontend.md に影響するもの

- AppState のフィールド追加・削除
- 新しい Context / hooks / reducer の追加
- 状態管理手法の変更（Context → URL params など）
- コンポーネント構成の大きな変更
- MapLibre / PMTiles 統合の変更（projection 切替、タイル URL 組み立て）
- 主要な画面フロー（年選択、領土クリックなど）の挙動変化

### worker.md に影響するもの

- エンドポイントの追加・削除
- キャッシュ戦略の変更
- R2 バインディングの変更
- CORS 設定の変更
- 環境変数の追加・削除
- 配信挙動の変化（Range Request 対応など）

## 反映不要の変更

- 振る舞いを変えない純粋なリファクタ（関数分割、変数リネームなど）
- テストの追加・修正
- ロギング・エラーメッセージ文言の調整
- 依存バージョンの上げ下げで、ドキュメントに書かれたインターフェースが変わらないもの
- フォーマット・lint 修正
- コメントの追加・修正

## 判断に迷う場合

- diff 行数が多くても、性質上 "内部構造の微調整" なら反映不要
- diff が小さくても、データフローや外部依存の変更なら反映する
- 1 コミットが複数ドキュメントに跨る場合は両方に反映する（例: 新しい中核型追加 → overview + 該当アプリ）

## スキップ判定の記録

反映不要と判定したコミットは、承認時にユーザーへ伝えるため、
ハッシュ + 理由（"純リファクタ" / "テストのみ" / "フォーマット" など）を軽くメモしておく。
全コミットをスキップしたファイルは乖離レポートで「反映すべき変更なし」と明記する。

## スキル自体の前提が変わる変更

以下の種類の変更は、このスキルのリファレンスファイル自身（change-impact.md / templates.md）の
改訂も必要になる。ドキュメント内容の更新とは別枠で「メタ変更」として乖離レポートに報告し、
自動修正はせず、ユーザーに skill 本体の改訂を依頼する。

- 新しいアプリディレクトリの追加・既存アプリの削除・統合・分割
  → templates.md に新ファイルのテンプレを追加／不要なテンプレを削除
- 中核の型の共有関係の変化（共有パッケージ化、アプリ別に完全分離）
  → overview.md の「中核の型」セクション定義を見直す
- 新しい外部サービス・配信基盤の導入（例: worker を廃止して別 CDN に移行）
  → 影響マップの節構成を書き直す
- `docs/` 配下の命名規則やファイル分割戦略そのものの変更
  → SKILL.md の「ドキュメントの構造」節、templates.md のセクション構成を書き直す

判断の目安: 「既存の影響マップに分類できない変更」「テンプレートの章立てが現構造と合わなくなる変更」は
メタ変更候補として立ち止まる。

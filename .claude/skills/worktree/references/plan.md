# 並行開発プラン

複数イシューの依存関係を分析し、安全に並行開発できるグループ（Wave）に分類する。

## 引数

- イシュー番号のリスト（例: `158 159 160 161`）

## フロー

### 1. イシュー情報を一括取得する

各イシューについて:

```bash
gh issue view <番号> --json title,body,labels,state
```

### 2. 依存関係を抽出する

各イシュー本文から以下の2種類の依存情報を読み取る:

- **明示的依存**: 先行イシューへの言及（例: `#158 の後に実施`、`依存: #157`）
- **暗黙的依存**: 変更対象ファイルの重複から推定されるコンフリクトリスク

変更対象ファイルはイシュー本文から読み取る。明記されていない場合は、
イシューの内容からコードベースを探索して推定する。

### 3. ファイル衝突リスクを検出する

対象ファイルの重複を検出し、コンフリクトリスクを評価する:

| リスク | 条件 |
|--------|------|
| 高 | 同じファイルを3つ以上のイシューが変更 |
| 中 | 同じファイルを2つのイシューが変更 |
| 低 | ファイル重複なし |

### 4. Wave に分類する

以下のルールで Wave（並行実行可能なグループ）を決定する:

1. 明示的な依存関係を優先する
2. ファイル衝突リスクが「高」のイシュー同士は同じ Wave に入れない
3. ファイル衝突リスクが「中」のイシュー同士も原則分ける（影響が軽微な場合は例外を検討）
4. 依存先イシューは必ず先の Wave に配置する

### 5. 結果を出力する

以下を出力する:

- Wave 分け（各 Wave に含まれるイシューとその理由）
- コンフリクトリスクマップ（注意すべきファイル衝突）
- 各 Wave 内の推奨マージ順（影響範囲が小さいものから）
